# Publishing Generated Output (Clients)

This guide focuses on the end-user workflow: how to publish the **client libraries** generated by `cdd-kotlin` and how to keep those libraries continuously up-to-date with your backend's OpenAPI specification.

## Synchronizing with the OpenAPI Server

The primary value of code generation is staying synchronized with the backend contract. A common and robust way to achieve this is via a scheduled CI/CD job (like a GitHub Actions Cron) that periodically fetches the latest OpenAPI spec, regenerates the client, and automatically opens a Pull Request if changes are detected.

### Automating with GitHub Actions

You can configure a GitHub Actions workflow in your client's repository (or a monorepo) that fetches the server's spec (e.g., from an API gateway or raw GitHub file) and regenerates the client.

Create `.github/workflows/sync-openapi.yml`:

```yaml
name: Sync OpenAPI Client

on:
  schedule:
    # Run at 02:00 every day
    - cron: '0 2 * * *'
  workflow_dispatch: # Allow manual triggers

jobs:
  generate-client:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout Client Repository
        uses: actions/checkout@v4
        with:
          path: client-repo

      - name: Set up JDK 17
        uses: actions/setup-java@v4
        with:
          java-version: '17'
          distribution: 'temurin'

      # Step 1: Download the latest OpenAPI Spec from your server
      - name: Fetch OpenAPI Spec
        run: |
          curl -H "Authorization: Bearer ${{ secrets.API_TOKEN }}" 
               -o client-repo/spec.json 
               https://api.yourcompany.com/v1/openapi.json

      # Step 2: Regenerate the Client using cdd-kotlin
      - name: Run cdd-kotlin Generator
        run: |
          # Assuming you have a cdd-kotlin executable or Gradle task configured
          cd client-repo
          ./gradlew run --args="from_openapi -i spec.json --clientName MyApiClient --output generated-src"

      # Step 3: Check for changes and create a PR
      - name: Create Pull Request
        uses: peter-evans/create-pull-request@v6
        with:
          path: client-repo
          token: ${{ secrets.PAT }} # Use a Personal Access Token to trigger downstream actions
          commit-message: "chore: Sync OpenAPI client with latest backend spec"
          title: "Automated Client Sync"
          body: |
            Automated PR to synchronize the Kotlin Multiplatform client with the latest `spec.json`.
            Please review the generated changes carefully to ensure backwards compatibility.
          branch: chore/sync-openapi
          base: master
          labels: "automated,openapi"
```

*This process ensures that whenever the backend API changes, a Pull Request is automatically generated containing the exact Kotlin source code changes required to support it.*

## Publishing the Generated Client

Once the generated code is merged into the `master` branch (e.g., after the PR from the cron job is approved), it needs to be packaged and published to a repository so other applications (Android apps, iOS apps, Spring Boot backends) can depend on it.

### Publishing Kotlin Multiplatform Libraries

Because `cdd-kotlin` generates a Kotlin Multiplatform project (KMP), publishing it involves uploading multiple artifacts (JVM, Android AAR, iOS Frameworks/Klibs) to Maven Central or a private repository like GitHub Packages or JFrog Artifactory.

#### 1. Gradle Setup for the Generated Client

In the generated client's `build.gradle.kts`, ensure the Maven Publish plugin is applied to the Kotlin Multiplatform extension:

```kotlin
plugins {
    kotlin("multiplatform") version "2.2.21"
    `maven-publish`
}

kotlin {
    // Define your targets (generated by ScaffoldGenerator)
    jvm()
    androidTarget { publishLibraryVariants("release") }
    iosX64()
    iosArm64()
    iosSimulatorArm64()

    // Required for multiplatform publishing
    explicitApi()
}

// Publish to an internal repository (e.g., GitHub Packages)
publishing {
    repositories {
        maven {
            name = "GitHubPackages"
            url = uri("https://maven.pkg.github.com/YourOrganization/YourRepository")
            credentials {
                username = System.getenv("GITHUB_ACTOR")
                password = System.getenv("GITHUB_TOKEN")
            }
        }
    }
}
```

#### 2. Release Automation

Create a separate GitHub Action in your generated client repository to automatically publish new releases when a new tag is pushed.

Create `.github/workflows/publish-client.yml`:

```yaml
name: Publish KMP Client

on:
  push:
    tags:
      - 'v*' # Trigger on version tags, e.g., v1.0.0

jobs:
  publish:
    runs-on: macos-latest # macOS is required to build and publish iOS native targets
    permissions:
      contents: read
      packages: write # Needed for GitHub Packages

    steps:
      - name: Checkout
        uses: actions/checkout@v4
      
      - name: Setup JDK
        uses: actions/setup-java@v4
        with:
          java-version: '17'
          distribution: 'temurin'
          
      - name: Publish all Multiplatform Targets
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        run: ./gradlew publishAllPublicationsToGitHubPackagesRepository
```

With this setup, the lifecycle of your client is fully automated:
1. Server changes API.
2. Cron job detects change, runs `cdd-kotlin`, and opens a PR.
3. Reviewer approves and merges PR.
4. Maintainer creates a GitHub Release/Tag (`v1.2.0`).
5. Publish Action builds all KMP artifacts (Android, iOS, JVM) and uploads them to your artifact registry.
